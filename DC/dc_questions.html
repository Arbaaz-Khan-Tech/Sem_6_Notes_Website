🔥Very Frequent Questions

Issues in Distributed Systems
  • Transparency: Access, location, concurrency, replication, failure, mobility.
  • Scalability: Challenges with size, geographical distribution, and admin control.
  • Fault Tolerance: Detecting, masking, recovering from failures.
  • Security: Authentication, confidentiality, integrity.
  • Concurrency & Resource Management
Ricart-Agrawala Algorithm
  • Type: Decentralized mutual exclusion
  • Uses timestamped REQUEST messages
  • CS entry: Sent REQUEST to all & received REPLY from all
  • Message Complexity: 2 × (N – 1)
Remote Procedure Call (RPC)
  • Allows client to invoke procedures on remote systems.
  • Components: Client Stub, Server Stub, RPC Runtime
  • Steps:
    1. Client calls local stub.
    2. Marshals request.
    3. Server stub receives and executes.
    4. Sends back result.
  • Transparent to programmer.
Load Sharing vs Task Assignment vs Load Balancing
  • Load Sharing: Distributes tasks to avoid idle resources.
  • Task Assignment: Static decision; maps tasks to processors before execution.
  • Load Balancing: Dynamic distribution to maintain equal workload.
Data-Centric Consistency Models
  • Strict: Read returns most recent write.
  • Sequential: Result of any execution is same as if ops were done in sequence.
  • Causal: Writes that are causally related must be seen in order.
  • FIFO: Writes from a process seen by all in order sent.
  • Weak/Eventual: No guarantees on real-time ordering.
Code Migration
  • Moving code between systems for efficiency/security.
  • Types:
    • Strong Mobility: Moves code + execution state.
    • Weak Mobility: Moves only code.
  • Examples: Applets, mobile agents.
DFS Features + File Service Architecture
  • Features: Transparency (location, access, concurrency), reliability, scalability, fault-tolerance.
  • Models: Remote Service Model, Upload/Download Model.

🔁Frequently Appeared Topics

Suzuki-Kasami Algorithm
  • Token-based mutual exclusion. Only one token exists.
  • Requesting process broadcasts request.
  • Holder sends token if not using CS.
  • Efficient: Message Complexity O(1) in absence of contention.
Maekawa's Algorithm + Quorum Set
  • Each process has a voting set; must get permission from all in set.
  • Quorum sets intersect (for safety).
  • Reduces message complexity from O(N) to O(√N).
  • Can lead to deadlock — needs extra mechanism.
Election Algorithms
  • Bully: Process with highest ID wins. Sends ELECTION messages to higher IDs.
  • Generic: Process initiates election. Candidate with best (lowest/highest) ID wins.
Logical vs Physical Clocks
  • Physical Clocks: Real-world time (NTP).
  • Logical Clocks: Lamport: If a → b, then L(a) < L(b). Vector Clocks: Tracks causality more precisely.
Client-Centric Consistency Models
  • Suitable for mobile/disconnected systems.
  • Models:
    • Monotonic Reads: Never see older data after seeing new.
    • Monotonic Writes: Writes follow order.
    • Read Your Writes
    • Writes Follow Reads
Process vs Thread
  • Process: Independent execution, own memory space.
  • Thread: Lightweight; shares memory with other threads in process.
  • Threads are faster, better for parallel tasks.
Global Scheduling Features
  • Centralized vs Distributed
  • Static vs Dynamic
  • Preemptive vs Non-Preemptive
  • Goals: Load balancing, minimizing execution time, fairness.
Message Communication Models
  • Transient: Message lost if receiver not ready.
  • Persistent: Stored in buffer until delivery.
  • Synchronous vs Asynchronous
Andrew File System (AFS)
  • DFS with client-side caching.
  • Whole-file caching
  • Callback mechanism for consistency.
  • Scalable and efficient.
Fault Tolerance
  • Techniques:
    • Replication: Active/passive
    • Checkpointing & Rollback
    • Redundancy: Hardware/software
  • Goals: Continue operation in presence of faults.
